## File Inclusion Vulnerability

* As the name says, it’s a vulnerability during the file process.
* This vulnerability has two methods: Local File Inclusion (LFI) and Remote File Inclusion (RFI).
* Local File Inclusion (LFI) vulnerabilities allow an attacker to read local files on the web server using malicious web requests.
* LFI can also be used for Remote Code Execution (RCE). In most cases, this is due to poor, missing, or unsanitized input sanitization.

### Directory Traversal

#### Understanding Vulnerability Checklist with Example of Path / Directory Traversal

#### What & Why Path/Directory Traversal

* Going back to an unwanted directory, or backward/forward directory, and it can be read.
* An attacker is able to read files on the server through the web application.

#### Summary

* Imagine having an online store that shows pictures of items for sale. When you click on an item, the website loads an image using a link like this:
  * `<img src="/loadImage?filename=218.png">`
* This link tells the website to fetch and display an image with the filename `218.png`. These images are stored on the server in a specific location, like `/var/www/images/`. The website combines this base directory with the requested filename to find and show the image. However, the issue is when the file name is user controllable on the client side or not validated by the server.
* If the web application doesn't have proper security measures, then the attacker can manipulate the filename parameter to trick the web application into showing files it shouldn't.
* For example, instead of a regular image filename, the attacker might use:
  * `https://insecure-website.com/loadImage?filename=../../../etc/passwd`
* The `../../../` part means "go up three levels in the directory structure." So, instead of going to the images directory, it goes up to the root directory of the server and looks for a file called `passwd` in the `/etc/` directory.
* On a Unix-based system (like Linux), `/etc/passwd` is a file containing user details. The attacker could access and retrieve this file, gaining information about the server's users.
* The same idea applies to Windows servers, but the path traversal sequence looks a bit different:
  * `https://insecure-website.com/loadImage?filename=..\..\..\windows\win.ini`
* The `..\..\..\` means "go up three levels" on a Windows system. The attacker could potentially access sensitive files like `win.ini`.

### Conditions

* Relevant Action:
  * Objective: The attacker aims to get sensitive files or perform actions on the server that could lead to unauthorized access or damage.
  * Example: Loading files or executing commands on the server by manipulating file paths in requests (e.g., reading system files, accessing user data, or executing arbitrary commands).
* Handling:
  * Method of Attack: The attacker manipulates input parameters, typically within URLs or input fields, to trick the application into accessing files or directories it shouldn't.
  * Example: Modifying a filename parameter in a URL to include directory traversal sequences like `"../"` to navigate up the directory structure.
* Unpredictable Action Situation (Defences and Vulnerabilities):
  * Defence Mechanism: Input validation and proper handling of user-supplied input to ensure that only allowed characters and paths are processed.
  * Example of Defence: Implementing a whitelist of allowed characters and preventing the use of `"../"` or other directory traversal sequences in user input.
  * Vulnerability: Lack of proper input validation and sanitization, allowing attackers to input special sequences that lead to directory traversal.
  * Example of Vulnerability: If the server application doesn't check or sanitize user-supplied input, an attacker might inject `"../"` to navigate to unintended directories.
* Unpredictable Action Situation (Defences and Vulnerabilities):
  * Defence Mechanism:
  * Input validation and sanitization: Besides validating the characters in the input, it's crucial to validate the entire input structure, ensuring it adheres to expected patterns.
  * Use a whitelist for file extensions: Define a list of allowed file extensions and validate that the provided input matches a safe and expected extension.
  * Example of Defence: If the application expects image files, only allowing extensions like `.jpg`, `.png`, or `.gif` and rejecting any attempt to input additional characters or directory traversal sequences.
  * Vulnerability: If file extension validation is weak or non-existent, an attacker might exploit it to append additional directory traversal sequences or execute malicious actions.
  * Example of Vulnerability: Allowing an input like `../../../etc/passwd.png` could bypass simple input validation that checks for directory traversal but fails to recognize the manipulation of file extensions.

### BlackBox Testing

* Map all the application:
  * Identify all instances where the web application appears to contain the name of a file or directory.
  * Identify all functions in the application whose implementation is likely to involve retrieval of data from a server filesystem.
* Test identified instances with common directory traversal payloads.

### Preventing Vulnerabilities

* Primary defence
  * Input validation & sanitization.
  * Employing access controls to restrict file and directory access.
  * Running the web server and associated processes with minimal necessary permissions to limit potential damage.
* Additional defence
  * Content-Type Validation: Consider validating the content type of the file rather than relying solely on the file extension. For example, checking that an uploaded file is truly an image rather than just relying on the `.png` extension.
  * Implement a whitelist approach for input validation, allowing only expected characters and blocking any attempts at directory traversal.
  * Use secure APIs and libraries that handle file operations safely.

#### Local File Inclusion (LFI)

* Attackers can execute malicious code and output it within the web application in an unsanitized way.
* It loads the file locally.

#### Remote File Inclusion (RFI)

* Similar to LFI but involves getting remote files of a web application.
* Remote file inclusions is similar, but the attacker is taking advantage of the web server's ability to call local files, and using it to upload files from remote servers. These remote files can be malicious code that executes in the context of the web server user.
* LFI + directory traversal: inclusion of a local file (in the web server directory or not) by "climbing" the server tree with `../` (relative path)
  * DVWA Linux system, test display `/etc/passwd` by moving back 5 directory levels:
  * `http://host/?page=../../../../../etc/passwd`
  * Even if this doesn't work, it doesn't mean that the website is immune to path traversal.
  * When filtering input, developers will often prevent the use of forward slashes, but not backslashes or encoded characters.

**Potential Vulnerabilities:**

* Despite input validation, potential vulnerabilities can still exist, especially concerning handling backward slashes.
