## Kerberoasting

### Kerberos

* Kerberos is a Network Authentication Protocol that uses tickets to verify the identity of users and machines, typically operating over both TCP and UDP Port 88.
* Kerberos is named after the three-headed dog from Greek mythology.
  * The client, the server, and the Key Distribution Centre (KDC) design to protect application access.
* Kerberos is the default authentication protocol used in Active Directory Domain Services (AD DS) environments.

Basic:

* Client: A user or machine that wants to access a service within the network.
* Service: The resource or application the client wants to access, which requires authentication.
* KDC (Key Distribution Centre): An Authenticator (the Heart of Kerberos), responsible for authenticating user and issuing Ticket Granting Tickets.
* How it works: When a client wants to access a service, it must first be authenticated by the KDC, which has two main functions/server.
  * Authentication Server/Service: Verifies the user and issue Ticket Granting Ticket (TGT) if the user authenticated.
    * Ticket Granting Ticket (TGT): A Ticket received after being authenticated and used to request the TGS.
  * Ticket Granting Service (TGS): The Client present the TGT to TGS which then issues a service ticket that allows the client to access the requested service.
* Note: There are a lot of messages between user and authentication server and Ticket granting server where some messages are in Plain text and some are encrypted.
* Types of Messages
  * Authentication Message: Contain the session key of the client and the server.
  * Tickets: Encrypted information like ClientID, ServiceID, Timestamp, TTL Etc.
  * Service Principal Name(SPN): A service finder which connected the client to the requested service.

### AS\_REQ (Authentication Service Request)

Message from User to Authenticated server

User's Initial Request (AS\_REQ):

* User -> KDC (AS): "Hi KDC, I need a TGT to authenticate to services. Here are my credentials (encrypted with my long-term key)."
  * The user sends an Initials request to Authentication Server in the Key Distribution Centre to obtain a TGS.
* Packet: The packet is encrypted with the user password and has timestamp to prevent replay attack.
* AS Verification: The Authentication Server verify the request by decrypting it using the user stored NTML Hash. If the credentials are valid the AS generate a TGT which is encrypted with the user NTML Hash and send back to user.

Wireshark AS\_REQ Analysis:

* pdata (Pre-authentication Data): Contains the timestamp and other data encrypted with the user's password, used to verify the user's identity.
* CNAME (Client Name): The username of the client requesting authentication.
* REALM: The domain name in which the user is requesting authentication.
* Etype (Encryption Types): The encryption types that the client supports, used to negotiate the encryption method with the KDC.

### TGS\_REQ (Ticket Granting Service Request)

Message from authentication server to user

User's Request for Service Ticket (TGS\_REQ):

* User -> TGS: "Here is my TGT and an authenticator to verify my identity. I want to access service SPN."
* TGT is used for requesting a service ticket.
* The Golden Ticket Attacks overwrite the TGT.

Decrypting

* TGS\_REQ Decrypting and Validation:
  * Decryption: The KDC (The TGS Server) start decrypting the TGS\_REQ using the TGT provided by client.
  * Service ID and authenticator validation: Its gets ServiceID from the decrypted TGS\_REQ and validate in the database with the authenticator.
* Session Key Decryption:
* TGT Session Key: The TGT Session key was used by client to encrypt parts of TGS\_REQ, The KDC uses the TGT to decrypt the TGT Session key.
* User Authentication Message: The TGS Session Key is then used to decrypt the user authentication message within the TGS\_REQ
* Creating Service Ticket:
* Service Secret Key: The KDC creates a service ticket that is encrypted with the service's secret key. This ensures that only the target service can decrypt and read the ticket.
* Service Ticket Contents: The service ticket contains information like the client's ID, a new session key for client-service communication, and the ticket's validity period.
* Note the user can’t decrypted the service secret key

### TGS\_REP (Ticket Granting Service Response)

The user decrypted the user authentication and send to service with service ticket.

TGS Response (TGS\_REP):

* TGS -> User: "Here is your service ticket (encrypted with the service's secret key) and a session key for communication with the service (encrypted with the TGS session key)."
* TGS Processing:
  * TGS decrypts the TGT using its secret key.
  * TGS decrypts the authenticator using the session key from the TGT.
  * TGS verifies the user’s identity and checks the request.

### Discretionary Access Control List (DACL)

* An Object DACL and System access control list are ordered collection of Access Control Entries.
* Everything in Active Directory is an OBJECT.
* No matter is group, user, computer, anything.
* Every object has its own access control. So, every user, group has its own Access control list.
* If those ACL is misconfigured or have unintended privileges which results in privileges escalation.
* By default, the advance feature is disable, enable it by going to Active directory user and computers. View >> Advance Feature
* In Properties of user >> Security Tab will give the object access control.

2. User: administrator

A Silver Ticket attack is another type of Kerberos ticket forgery attack like the Golden Ticket attack, but it targets service tickets (TGS tickets) rather than the Ticket Granting Ticket (TGT). In a Silver Ticket attack, an attacker forges a service ticket for a specific service, allowing them to access that service without needing a valid TGT from the Key Distribution Centre (KDC).
