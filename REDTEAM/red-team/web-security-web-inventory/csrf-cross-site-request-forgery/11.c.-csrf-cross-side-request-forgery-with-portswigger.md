| ### **What & Why CSRF** |
| ----------------------- |

* A CSRF attack is an attack where the attacker find vuln of original domain and insert malicious code and send to victim which leads the victim user to carry out an action unintentionally while the victim is authenticated.
* In short A CSRF use browser **default functionality of cookie** , Attacker use the **authenticated User and send Malicious request**.

| ### **CSRF Conditions** |
| ----------------------- |

* A Relevant Action : An Function which damage the victim/User.
* Cookie Based Session Handling : Cookie System is in Use.
* No Unpredictable Parameters : Any Unpredictable parameter like CSRF token which is randomly generate character form attacker Point of view.
* **Requirements of CSRF**

1. The attacker has to send a way to the victim that the victim user is going to click.
2. The victim user has to be already authenticated into the web application.
3. In order to define the all point it need to match CSRF Condition.

* CSRF Condition.
  * In Order for CSRF Attack to be Possible.
  * **A relevant Action :**- is a action where it would exploitable that cause efforts to victim user. / : something that is import and if compromise can lead to dangerous consequence
  * **Cookie Based session handling**: session cookie :- so the application has to be using cookie for session management and the reason it has to be using cookie is because this attack is buil on exploiting the default functionality in the browser which is to automate send cookie to the assign domain when a request is made by the user.
  * **No unpredictable request Parameters** :- that parameter which is not predictable by the attacker.
    * POV of attacker , example CSRF Token is an unpredictable.

| ### **Primary Defence** |
| ----------------------- |

* How should token be generated ?
  * Unpredicatable with high string similar to session token,
  * Tied to the user session tring so that attacker cant use it own generated CSRF.
  * Validate before the relevant action is executed before the email/request or any chance happen the application need to confirm the CSRF token validation and of user session are tied.
  * Token generated should not be transmiteed with cookie.
    * How the CSRF token be transmitted ?
    * Hidden filed of anu HTML form that submiteed using HTTP POST Method (Not Seen in source code).
    * Custom request header HTTP.
    * Token Submitted in the URL query string are less secure,
      * URL are logged in the backend and if the URLs its has sessions and token, any third-party website can gained access the different domain token.
    * Token generally should not be transmitted within cookie
    * The malicious JS code can ask cookie and redirected to attacker website.
    * HTTP injection or HTTP header Injection.

| ## **Additional Defence** |
| ------------------------- |

* **SameSite**
  * The **SameSite** cookie is an attributes which used to control whether cookie are submitted in **XSS Cross Site Request**.
  * It added in the cookie response header à When the server issue a cookie the attributes can given three value which is strict , LAX and None.
  * Its a Security Feature which makes website/server include Cookie. Without Cookie the server won’t Send request.
  * Broswer Send cookie in every request to the domain that issued them even if the request was trigger by the third-party domain.

| **Origin Site**                      | **SameSite**               |
| ------------------------------------ | -------------------------- |
| **https://cyber.hasanrhani.com:443** | **https://hasanrhani.com** |

**SameSite** : work on TLD \[Top Level Domain]

* Its has three attributes.
  * Strict.
  * LAX. (Defalutly Used by Chrome.)
  * None. (Defaulty used by all Broswer.)
* **Strict**
  * Strick mean the cookie will only send to the first party web and not send to the intiated by third party à if third party request the session is removed and server reject.
  * The cookie attributes is added by server when the request doesn’t come form any third-party domain.
  * Broswer will not send it in any Cross Site Scripting XSS request , it site request doesn’t matches the current site domain, if not then it will not include cookie parameter which result in “missing cookie value”.
* **LAX**
  * LAX is initiated form the third part website then the cookie will not be submitted.
  * Lax send XSS Request if the condition are met.
    * Request of the HTTP GET Method.
    * The request has to be navigated form top level like the user clicking a click.
  * It’s a rule like security guard.
    * When this rule is in place the guard watches and will allow the cookie to be shared with other websites but only in specific situations.
      * When Someone clicks on a links or goes to new Websites \[Redirections on Same Domain /Other Domains]using Browser in this case the Guard says OKAY to share the cookie Value.
      * When the person is looking at stuff on a website using the GET method like clicking or reading a Hacking news which has sources
* **None**
  * Checks for no Cookie.
  * Imagine you have a special cookie, and you're like a guard who decides when to share this cookie with websites.
  * When a cookie has "SameSite=None," it's like telling the guard to let anyone have the cookie, no matter what.
  * So, all web browsers will send this cookie to the website that gave it out, even if the request comes from a completely different website. No restrictions!
  * Usually, browsers follow this rule by default, except for Chrome.
  * Some cookies need to be shared this way, like tracking cookies, which are used for keeping track of what you do online.
  * But, this can also be risky because it can potentially expose sensitive information. That's why it's essential to use this carefully.
  * If you want to use "SameSite=None," you also need to make sure your website is using HTTPS, which is a secure way of communicating online. Otherwise, browsers won't accept the cookie.
  * So "SameSite=None" is like telling the guard to share the cookie with everyone, but you should only do it when you have a good reason and are using a secure connection.
